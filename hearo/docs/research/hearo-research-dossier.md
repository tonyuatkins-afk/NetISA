# HEARO research dossier: filling the six knowledge gaps

The most consequential finding is that **PnP BIOS enumeration from real-mode DOS is fully spec'd, well-defined, and almost trivially small to implement** — roughly 200 lines — and is the *only* path that reaches BIOS-PnP-only laptop chips like the YMF715 in your 320CDT. UNISOUND and SETYMF deliberately do not implement it (their authors say so on VOGONS); Linux's `sound/isa/opl3sa2.c` does, and it gives you a complete, primary-source reference implementation. Beyond that, every other gap has the same structural answer: established DOS practice exists, was followed by every shipping audio driver, and is documented well enough to copy verbatim. The Yamaha chip's hostility toward Sound Blaster Pro emulation, the Watcom `__STK` trap into INT 21h, and the "Cannot load COMMAND" wedge are all instances of the same root failure mode — assumptions that hold on real Creative silicon, on faithful emulators, or on neutral compilers, but break specifically when a clone, a stack-checked function, or an in-flight DMA leak finds the seam. The recommendations below are calibrated to that reality.

## 1. PnP BIOS enumeration is the only way to find your control port

The Plug-and-Play BIOS Specification 1.0A (Compaq/Phoenix/Intel, 5 May 1994) defines a **far-call entry point** discovered through a `$PnP` signature structure in ROM, *not* an INT 1Ah service. You scan `F0000h–FFFF0h` on 16-byte boundaries for the ASCII `$PnP` signature; the structure is 33 bytes; the byte at offset 5 is the length, the byte at offset 8 must make the length-byte sum zero, and offsets `0Dh:0Fh` give you a real-mode `offset:segment` to far-call. **The single critical detail** missed by most informal write-ups is offset `1Bh`, the "BiosSelector" — the real-mode 16-bit data segment the BIOS expects you to push as the last argument of every call. Calling convention is C/cdecl: arguments pushed right-to-left as 16-bit words, function number pushed last, `cdecl` cleanup, return code in AX (0 = success, bit 7 set = error), all other registers and FLAGS preserved. You must guarantee at least 1024 bytes of stack.

Three function calls do the entire job. **Function 00h** ("Get Number of System Device Nodes") returns the count and the maximum node-buffer size. **Function 01h** ("Get System Device Node") fills a buffer for a given handle and updates the handle to the next one — pass `Control = 0x0001` to request the *current dynamic* configuration (the BIOS-assigned resources you actually need), and stop when the handle returns `0xFFh`. The buffer's first 12 bytes are a fixed header; the bytes at offsets 3–6 are the **EISA-compressed Device ID**; everything after offset 12 is a tag-stream of resource descriptors using the same format defined in the ISA-PnP spec.

The compressed ID encoding packs three uppercase letters (A=1, 5 bits each) into the high bits of the first two bytes, and three hex digits + revision nibble into bytes 2–3. **`YMH0021` encodes as the byte sequence `65 A8 00 21`**, and that is the per-device handle you match against. The card-level ID `YMH0030` (which UNISOUND and SETYMF look for via ISA-PnP isolation) is *not* what BIOS enumeration sees — BIOS exposes the logical device, not the card. The Linux kernel's `pnp_device_id` table at `sound/isa/opl3sa2.c` confirms this exactly: its `snd_opl3sa2_pnpbiosids[]` (PnP-BIOS path) lists `YMH0021` and `NMX2210`, while the separate `snd_opl3sa2_pnpids[]` (ISA-PnP isolation path) lists `YMH0020/0030/0800/0801` and `NMX2200`. The driver explicitly logs `pnp_device_is_pnpbios(pdev) ? "BIOS" : "ISA"` to distinguish the paths, which is your primary-source confirmation that **BIOS enumeration is a real, separate, working code path** for this exact chip.

Inside the YMH0021 node's allocated-resource block, walk the small-tag/large-tag stream (high bit clear = small tag, `(byte>>3)&0xF` = tag name, `byte&7` = data length) until you hit `END_TAG = 0x78`. Tags `0x47` (I/O port range descriptor, 7 data bytes) and `0x4B` (fixed-location I/O, 3 data bytes) carry the I/O ranges. Linux's mapping is the contract: `pnp_port_start(pdev, 0..4)` returns SB Pro base, WSS base, FM (OPL3, typically `0x388`), MPU-401, and finally **the OPL3-SA3 control base — the fifth I/O resource (index 4)**. The control region is 2 bytes wide (an index port and a data port), which lets you sanity-check that you've parsed the right descriptor when ordering looks wrong: it is the only 2-byte window in the high range. Linux **always** validates whichever path discovered the chip with a port-level probe of the MISC register (index `0x0A`) before trusting it — and HEARO should do the same.

**The Linux probe is also your fallback discovery mechanism** when BIOS enumeration fails or returns something implausible. Try candidate control bases in order: `0x370, 0x100, 0x538, 0xE80, 0xF86`. The 320CDT and other Toshiba laptops are documented to use `0x100`, contradicting the common assumption of `0x370`, so **never hardcode**. The probe sequence: read `MISC` at `index 0x0A`; if `0xFF`, no chip. Toggle the low three bits of MISC and confirm read-back; restore. Then probe MIC register at `index 0x09`: write 0x8A, confirm `(read & 0x9F) == 0x8A`, restore to 0x9F. The **chip-version code is `MISC & 0x07`**: 1 = OPL3-SA2 (YMF711), 2 = OPL3-SA3 (YMF715), 3 = YM715B, 4 = YM719 (marketed as SA4), 5 = OPL3-SA3 in Toshiba Libretto 100, 7 = NeoMagic MagicWave. The "password" register write of `0x1D` that older `oss/opl3sa.c` performs at base `0xF86` is a vestige of YMF701 and is `#if 0`-ed out in modern ALSA — you do not need it on YMF715.

**What couldn't be verified.** No source code for SETUPSA, SETYMF, or UNISOUND is public; their authors have confirmed in VOGONS threads that they don't open-source them. Tiido (SETYMF) confirmed his tool "is not really meant to handle EEPROMless configuration mode that a laptop can make use of" — exactly your case. JazeFox (UNISOUND) confirmed "UniSound does not use BIOS calls (it does direct ISA communication only)." This is the gap HEARO is filling; **no public DOS reference implementation exists**, which is why the spec-from-first-principles approach is correct.

**Recommendation for HEARO.** Implement a four-step cascade matching Linux: (1) PnP BIOS scan → enumerate → match `YMH0021` → parse 5th I/O = control base, also harvest SB/WSS/FM/MPU/IRQ/DMA; (2) ISA-PnP isolation as a fallback for desktop YMF715 cards on a real ISA bus, matching card IDs `YMH0020/0030/0800/0801`; (3) port probe of `{0x370, 0x100, 0x538, 0xE80, 0xF86}` using the §1 probe; (4) **regardless of how the base was discovered, always validate with the MISC-register probe** before initializing. This is exactly what the kernel does and exactly fills the gap UNISOUND/SETYMF cannot.

## 2. Cooperative multitasking: ISR sets a flag, foreground pumps

The canonical DOS audio architecture from SB 2.0 onward is **single contiguous DMA buffer in low memory + auto-init DMA + DSP block size set to half the buffer**, which causes the SB DSP to fire IRQ at half-empty and again at end. The Sound Blaster Series Hardware Programming Guide states this verbatim ("the application allocates a stationary DMA buffer which must not straddle a 64 KB physical page boundary; the DSP block transfer size is then set to half of the DMA buffer size"), and Tubetime's reverse-engineering of SB1.0 firmware confirms that the IRQ is the *DSP's* trigger, not the DMA controller's — the DMA controller sees one buffer that wraps. **A true two-buffer ping-pong from DMA's perspective is not the standard pattern**; it adds complexity for no acoustic gain.

The "ISR sets `refill_pending`, foreground calls `pump()`" pattern is the C-friendly variant of this and is what OpenCP uses (with an additional software pre-mix buffer ahead of the DMA buffer, parameterized as `mixbufsize` versus `plrbufsize`). Impulse Tracker takes the opposite approach (mixing inside the ISR in pure asm), but that only works because Lim hand-managed every stack frame; it is not advisable in C. **The ISR's job is six instructions**: increment a missed-refill counter if `refill_pending` is already set (diagnostic), set `refill_pending = 1`, toggle `refill_half`, ack the DSP IRQ (`IN` from `base+0x0E` for 8-bit, `base+0x0F` for 16-bit), send EOI to the PIC (slave first if IRQ ≥ 8), and `IRET`. Single-byte writes are atomic on x86, so the flag does not need `cli/sti` for a simple test. **However**, the test-and-clear sequence in the foreground must wrap in `_disable()/_enable()` because you read both `refill_pending` and `refill_half` and need a consistent pair. Critically, do *not* hold interrupts off during `mixer_render()` itself — that defeats the whole point of the pump pattern and produces classic DOS audio buzz.

The Watcom `__STK` reentrancy bomb is the most dangerous part of this and your quality gate caught it correctly. Watcom emits a call to `__STK` at the start of every function compiled with stack checking enabled (the default), to verify stack space. The overflow path in `bld/clib/startup/a/stkrdu.asm` reaches `__STKOVERFLOW`, which **calls INT 21h AH=4Ch (and prints a string via DOS) — INT 21h from whatever context overflowed**. If that context is your ISR or any function transitively reachable from your ISR, you have re-entered DOS while `InDOS` may be non-zero, and the system corrupts itself. The fix is documented in QNX/Watcom and embedded in Open Watcom's own `bld/src/goodies/mouse.c` example: **put every ISR and every function it can call in its own translation unit, compile that unit with `-s -zu` (`-s` disables stack checking → no `__STK` emission, `-zu` declares SS≠DS), and bracket the source with `#pragma off (check_stack)`**. Lock the resulting code and data with DPMI 0x600 if running under any DPMI host. Pre-allocate any stack the ISR uses; never call libc, floating-point, or DOS from within it. **Do not call `getch()`/`kbhit()` from libc**; use INT 16h AH=01h (peek) and AH=00h (read) directly — those are pure BIOS, never set `InDOS`, and are fast.

For the case where `pump()` is too slow, three mitigations stack: enlarge the buffer (44.1 kHz/16/stereo at 16 KB total = ~93 ms per half, double it for safety on a 386), pre-mix into a software ring so `pump()` only `memcpy`s into the DMA half (the OpenCP approach), and bound foreground work per loop so a directory enumeration becomes a chunked yield-friendly operation that calls `pump()` between batches. **Treat `pump()` as your cooperative yield function**; anything taking more than a few milliseconds in the foreground must call it. Surface `refill_overruns` as a runtime diagnostic — non-zero under normal browsing means buffer too small or pump too slow. The recommended main-loop skeleton is `pump(); if (kbhit_int16()) handle_key(); redraw_if_dirty(); pump();` with the trailing `pump()` as belt-and-braces after a heavy redraw.

The Apogee Sound System / DMX / Miles model (timer-driven mixer in IRQ) is *louder* in retro game lore but worse for HEARO's case, because your foreground does file browsing and decoder header parsing, both of which fight an IRQ-driven mixer for cache and stack. **The flag-and-pump model is correct for a UI-driven player.**

## 3. Tracker correctness: the OpenMPT wiki is the bible

There is no single document called "the tracker format bible." The closest equivalents are three layered resources. **OpenMPT's player_tests** is the authoritative regression suite — `https://wiki.openmpt.org/Development:_Test_Cases/{MOD,XM,S3M,IT}` indexes 23 MOD tests, ~30 XM tests, 22 S3M tests, and **123 IT tests** (OpenMPT itself passes 122/123). Test modules download from `https://resources.openmpt.org/player_tests/{mod,xm,s3m,it}/`. The maintainers explicitly state "**bit-exact output is not the goal, correct playback is**"; tests are designed to be auditioned at 44.1 kHz with HEARO's output on the left channel and the reference tracker's pre-rendered output on the right, listening for phase identity. **Schism Tracker's "Player Abuse Tests"** (a tarball linked from `https://github.com/schismtracker/schismtracker/wiki/Player-abuse-tests`) provides ~25 IT modules paired with reference WAV files generated directly from IT 2.14p5's disk writer, focused specifically on the most-misimplemented IT effects. **The OpenMPT wiki page `Manual:_Compatible_Playback`** is the single most useful read in the whole tracker world: it enumerates every behavior OpenMPT had to add a compatibility flag for to match Impulse Tracker, FastTracker 2, or Scream Tracker 3 — i.e., it is a list of all the bugs your decoder needs to faithfully reproduce.

The most-misimplemented effects per format, drawn from the test suite descriptions and libxmp's `tracker_notes.txt`, fall into recognizable patterns. **MOD/ProTracker** has the 9xx (sample offset) double-add bug in PT1/2 that PT3+ fixed; vibrato that does not advance on the first tick *and* does not apply offset on tick 0; arpeggio that wraps to period 0 above the Amiga range; instrument-swap deferred to end-of-loop on PT1/2; and finetune-dependent period clamping. **S3M** uses *one* shared "last non-zero parameter" memory across most effects (Dxy, Kxy, Lxy, Exx, Fxx, Ixy, Jxy, Qxy, Rxy, Sxy) — the most common bug is per-effect memory; combined slides (Kxy/Lxy) silently skip the first tick on ST3-saved files; offset wraparound differs between GUS and SB drivers (detect via `Int:Gp` field); and the Adlib zero-volume-note bug suppresses the next note. **XM/FastTracker 2** has E5x finetune semantics utterly different from MOD; envelope position is incremented *before* point evaluation (same as IT); volume-column vibrato stacks with effect-column vibrato to advance the waveform at *double* speed; multiple EEx pattern delays count only the *rightmost* (opposite of S3M). **IT/Impulse Tracker** has the Compatible-Gxx flag controlling whether Exx/Fxx/Gxx share memory; resonant lowpass filter requiring Jeff Lim's coefficients in careful integer math (sloppy implementations produce audible "scratching" on `extreme-filter-test-1.it`); bidi loops one sample shorter in IT's software mixer than on the GUS driver; and a long list of empty-sample-map / NNA / DCT / MIDI-macro edge cases.

**NNA support is mandatory for IT**, full stop. Hundreds of well-known IT modules rely on New Note Actions for polyphony management; an IT decoder without NNA is not "90% correct," it's just wrong. The resonant filter is similarly load-bearing. Everything else admits graceful degradation.

For **OPL3 General MIDI playback (MIDIFM)**, the canonical reference is `libADLMIDI` (`https://github.com/Wohlstand/libADLMIDI`), whose `banks.ini` indexes ~78 GM-OPL3 banks named after their original drivers (DMX bank 64, Apogee bank 67, AIL/Miles bank 0, HMI Descent banks 2–5, Fat Man 4-op bank 59, WOPL bank 68, MS-AdLib bank 77). A separate **volume model** must be paired with each bank because every classic driver had its own — pick one bank and one volume model, ship that, and accept that some MIDI files will sound "wrong" because **classic FM-MIDI was never standardized**. **A drum-length fixer is essential**: many MIDI percussion notes are shorter than the OPL3 ADSR can render audibly, so the sequencer (not the synth) must extend any drum note shorter than ~50 ms. The DMXOPL bank (`https://github.com/sneakernets/DMXOPL`) is widely regarded as the best-sounding free GM-OPL3 bank.

For **VGM**, restrict initial scope to YMF262 (OPL3) packs from `https://vgmrips.net/packs/`; supporting all chips is a multi-year project. The current spec is 1.71 at `https://vgmrips.net/wiki/VGM_Specification`; unknown chip types must be silently skipped per spec. For **WAV**, RIFF parsing for fmt 0x0001 (PCM 8/16-bit) and 0x0003 (float) is sufficient; honor the data chunk size; resample to mixer rate. For **MIDI**, implement SMF Type 0 and Type 1; Type 2 essentially never appears in the wild.

The pragmatic minimum bar for HEARO on a 386/486 budget: **pass roughly 80% of OpenMPT's MOD/S3M/XM tests, the common cases of Schism's IT abuse tests, MIDI Type 0/1 with libADLMIDI bank-68 or DMXOPL via the DMX volume model**. That is competitive with most DOS-era players and well above what libmodplug ever achieved.

## 4. Real Sound Blaster behavior: the YMF715 lies, the spec is incomplete

The Creative Sound Blaster Series Hardware Programming Guide is the canonical source but is incomplete on three behaviors that real hardware exhibits. **The DSP busy bit at port 2xCh has a free-running cycle on real SB16** (CT174x, CT2230, CT2940) that toggles even with no I/O activity at an unknown rate; on SB Pro and earlier the cycle only runs during active DMA. SB 2.0 returns specific bytes `0xAA` (busy) and `0x2A` (ready), not the documented `0xFF`/`0x7F` — your code must check **bit 7 only**, never the whole byte (Hysteria's *Overload* depends on this). DOSBox-X's `dsp write busy delay` documentation says >20 µs breaks IRQ-0 direct-DAC games and >1 ms causes lag, which calibrates real hardware to single-digit microseconds per write transaction (the DSP is an 8051 at ~12 MHz). The DiamondWare reference driver's recipe is **65,535 reads of port 2xCh as the worst-case timeout** — about 65 ms wall-clock at 8.33 MHz ISA — which is generous for any real card and any plausible clone. On SB16 specifically, the busy bit never stays clear, so the algorithm is "read up to 8–16 times for any bit-7-clear sample, then immediately write." For direct-DAC (cmd 0x10) on SB16, *do not* poll busy at all.

**High-speed mode (Time Constant > 0xD2) is a SB Pro / SB 2.0 phenomenon that SB16 silently obviates**. On real SB Pro 2 (CT1600/CT1690), the DSP firmware in high-speed mode literally devotes 100% of its 8051 CPU to servicing DMA and stops accepting any DSP commands except a reset (which does not return `0xAA` until you send the reset twice / wait correctly). **On SB16 (DSP 4.x), commands 0x90/0x91 are silently aliased to 0x1C/0x14**; the high-speed handshake is no longer needed — sample rates up to 44.1 kHz stereo work in normal mode using the new generic 0xB0–0xCF commands with `set sample rate` (cmd 0x41/0x42). Clones diverge here in a way that bites HEARO: ESS, OPTi, and YMF715 typically downgrade silently or behave unpredictably under high-speed; **never issue 0x90/0x91 if the DSP copyright string (cmd 0xE3) does not contain "Creative"**. The DSP version (cmd 0xE1) is unreliable as a clone gate by itself — many clones return DSP 3.01 (SB Pro 2) or even 4.x without implementing the corresponding feature set.

**The Yamaha YMF715 specifically reports DSP 3.01 (SB Pro 2)** but its SB Pro emulation is broken in several ways: the master-volume mixer register `0x22` doesn't scale correctly versus a real CT1345 (workaround: set `0x22` to `0xFF` and control everything via FM/Voice mixers `0x26`/`0x04`); reading back the stereo/filter bit at mixer reg `0x0E` returns nonsense bits (`0x11`-style) compared to real CT1345; the chip lacks the SB ADPCM commands `0x16/0x17/0x76/0x77` (8-to-2 and 8-to-3) — only 8-to-4 ADPCM works, same limitation as ESS AudioDrive; on VIA P4 chipsets it underruns or drops bytes when nag-polled; and on AudioTrix variants the audio routes through DB60XG analog inputs requiring an `INIT3DXG` SysEx, otherwise the card sounds mute. The OPL3-SA2 (YMF711/718) is more stable than the SA3 (YMF715/719) — VOGONS users actively recommend SA2 over SA3 for SB Pro use.

**Auto-init DMA exit via 0xDA / 0xD9** is reliable on real Creative hardware but anecdotally not on YMF715 / ESS — the only universally reliable stop sequence is a full DSP reset. DSP V4.13 (CT2950, CT4170) has a documented single-cycle "extra byte" bug fixed by setting the DMA controller to auto-init even when the DSP is in single-cycle mode. **CT4170 ViBRA16XV does not support proper HDMA**: if `BLASTER` H == D (both on the 8-bit channel), this is a Vibra16XV-class card and 16-bit playback runs over 8-bit DMA. The IRQ-acknowledge requirement is asymmetric: SB16 halts DMA until you read `2xE` (8-bit) or mixer reg `0x82` (16-bit); SB Pro and earlier ignore IRQ-ACK and play continuously. **The 16-bit DMA page registers are easy to get wrong**: channel 5 page is `0x8B` (NOT `0x83`), channel 6 is `0x89`, channel 7 is `0x8A`, with address ports `0xC4`/`0xC8`/`0xCC` and counts `0xC6`/`0xCA`/`0xCE` — and the count is in 16-bit words, not bytes, and the address is in 16-bit-aligned units (one 64K word page = 128K of physical RAM).

The mixer chips (CT1345 on SB Pro, CT1745 on SB16) have **no documented inter-write settle time**, but if an IRQ fires between the index write to `2x4` and the data write to `2x5`, an ISR that touches mixer ports leaves the index pointer in an unexpected state. Wrap mixer index/data pairs in `cli/sti`, or always re-load the index in any ISR that touches mixer registers.

The defensive detection cascade for HEARO is: reset DSP (bail with "no SB" if no `0xAA` within 65,535 polls); cmd 0xE1 → DSP version; cmd 0xE3 → copyright string; if string is empty/NUL or doesn't contain "CREATIVE", set `flag_clone = true`; if SB16, read mixer `0x80/0x81/0x82` and detect Vibra16XV via `H == D`. **On any clone, never use cmd 0x90/0x91 (use 0x1C/0xC0 instead), never use ADPCM commands 0x16/0x17/0x76/0x77, never trust mixer 0x22 master-volume scaling on YMF715, and probe with cmd 0xF3 (16-bit IRQ trigger) before trusting a clone's claim of DSP 4.x.** Graceful degradation order: SB16 16-bit native → SB16 16-bit on 8-bit DMA (Vibra16XV path) → SB Pro 8-bit stereo → SB 2.0 8-bit mono → SB 1.x single-cycle → direct DAC last resort.

## 5. DMA buffers: low memory, page-trim, no UMB

The 8237 page-boundary constraint is mechanically simple but its behavior is silent corruption, not an error: **the 8237's 16-bit address counter wraps within its 64K page; the external page register is not incremented**, so when an 8-bit DMA transfer crosses a 64KiB physical boundary, the controller "happily continues the transfer with whatever data it finds at the new address" (OS/2 Museum, the canonical reference). The 16-bit channels (5–7) on the slave 8237 count *words*, giving an effectively 128 KiB-aligned boundary in physical address. **The boundary check formulas are `(P >> 16) != ((P + N - 1) >> 16)` for 8-bit and `(P >> 17) != ((P + N - 1) >> 17)` for 16-bit DMA.** Some chipsets and EMM386 silently emulate a 24-bit address that *does* cross — you must not rely on this.

The right allocation strategy for HEARO is the simple "**allocate 2N bytes, find the page-aligned region, optionally shrink with INT 21h AH=4Ah**" pattern. The juj/64kalloc trick (free-and-reallocate exploiting Borland Turbo C's first-fit allocator) is clever but compiler-specific and not portable to Watcom. The Sound Blaster Programming Guide's own recommendation — half-buffer ping-pong inside one 64K page with a small (4–8 KB or up to 32 KB) buffer — is what every shipping driver does.

**Allocate in conventional memory below 640 KB.** UMB is dangerous because under EMM386 the C0000–EFFFF *linear* range is implemented by paging extended memory back below 1 MB; the page is physically above 1 MB, and ISA 8-bit DMA cannot reach above 1 MB. ISA 16-bit DMA reaches 16 MB but requires physical contiguity which UMB cannot guarantee. HMA is at 100000h–10FFEFh — physically above the ISA 8-bit DMA reach, period. **Set INT 21h AH=58h to first-fit-low (`AL=01h, BL=00h`) and unlink UMB (`AX=5803h, BX=0000h`) immediately before allocating the DMA buffer**, then restore both afterward, otherwise a system-wide setting may give you a UMB block.

**VDS (Virtual DMA Services, INT 4Bh)** exists but is broken in practice for buffers above ~1 MB+64 KB. EMM386, QEMM, and DOS extenders only translate VDS calls correctly within that low region; above it they return the input address unchanged while claiming success. The OS/2 Museum has multiple posts documenting this. **Don't trust VDS for extended-memory DMA buffers under bare DOS**; allocate in conventional memory and the linear address equals the physical address (DJGPP's recommendation: `__dpmi_allocate_dos_memory()` for the same reason).

The boundary-trim allocation for HEARO:

```c
unsigned alloc_dma_buffer(unsigned N_bytes, unsigned *out_seg) {
    unsigned paras = (N_bytes + 15) >> 4;
    union REGS r; r.h.ah = 0x48; r.x.bx = paras * 2;
    int86(0x21, &r, &r);
    if (r.x.cflag) return 0;
    unsigned base_seg = r.x.ax;
    unsigned long phys = (unsigned long)base_seg << 4;
    unsigned long page  =  phys           >> 16;
    unsigned long end   = (phys + N_bytes - 1) >> 16;
    if (page == end) { *out_seg = base_seg; }
    else { unsigned long aligned = (page + 1) << 16;
           *out_seg = (unsigned)(aligned >> 4); }
    return base_seg;  /* keep this for INT 21h AH=49h on exit */
}
```

Pad the buffer with a 16-byte sentinel (`DEADBEEF` pattern) on each side and validate it on exit — an in-flight DMA over-run will trash the sentinel before it trashes the next MCB header.

## 6. The "Cannot load COMMAND" wedge is DMA bleeding into freed memory

The Memory Control Block chain is an implicit-by-size structure of 16-byte headers: byte 0 is `'M'` (chain member) or `'Z'` (last), bytes 1–2 are the owner PSP, bytes 3–4 are size in paragraphs, bytes 5–15 are reserved (or the program name in DOS 4+). **DOS validates the signature byte on every memory operation**; if it's not `'M'` or `'Z'`, you get "Memory allocation error" and OS/2 Museum's verdict applies: "all bets are off because something in the system corrupted memory, and nothing can be trusted anymore." VOGONS thread t=76376 documents the *Might & Magic 1* case verbatim — a buffer too small for the data written into it, overwriting the following MCB header, producing exactly your symptom: "Memory allocation error, cannot load COMMAND.COM, system halted."

When a transient program exits, COMMAND.COM's resident portion (bottom of memory) checks a checksum on the transient portion (top of memory); if overwritten, it reloads from `%COMSPEC%`. The "Cannot load COMMAND" message appears when reload's `INT 21h AH=48h` fails — almost always because **the MCB chain is corrupt**. For a sound player, the dominant cause is **DMA writes after program exit**: the auto-init DMA controller is still asserting DREQ; it writes into freed memory after DOS has reallocated that paragraph to COMMAND.COM; the new MCB header gets stomped. The secondary cause is **ISR firing after vectors aren't restored**: the SB DSP IRQ vector points into freed code, the next interrupt jumps into garbage. DOSBox-Staging issue #396 documents Zone66 doing exactly this.

**The cleanup order on exit matters and there is exactly one correct sequence**, synthesized from the Sound Blaster Programming Guide, Quake's `snd_dos.c`, and OSDev's ISA_DMA page: (1) mask the DMA channel at the controller — `OUT 0x0A, channel|4` for 8-bit, `OUT 0xD4, (channel-4)|4` for 16-bit — *this stops the ISA bus from seeing DREQ regardless of what the DSP does*; (2) stop the DSP DMA engine (cmd `0xD0` pause 8-bit, `0xD5` pause 16-bit, `0xDA`/`0xD9` exit auto-init) *and then* full DSP reset (write 1 to `base+6`, wait, write 0, poll for `0xAA`); (3) disable the speaker (cmd `0xD3`); (4) ACK any pending IRQ (`IN base+0xE` for 8-bit, `IN base+0xF` for 16-bit, then EOI to PIC); (5) restore the SB DSP IRQ vector; (6) restore INT 1Ch / INT 8 timer hooks; (7) restore INT 9 keyboard hook; (8) re-enable interrupts; (9) **validate the MCB chain**; (10) free the DMA buffer with INT 21h AH=49h; (11) free other allocations; (12) exit via INT 21h AH=4Ch. The reason for the order is straightforward: masking DMA *before* DSP reset is critical because the SB DSP reset does *not* halt an already-issued ISA DMA cycle — the controller is independent. Restoring vectors *before* freeing memory prevents a spurious interrupt from jumping into the freed code's segment. Validating MCB chain *before* `AH=4Ch` lets HEARO surface a clean diagnostic ("HEARO detected MCB corruption — DMA cleanup failed") before DOS panics with the cryptic system-halt message.

The MCB chain walker is small enough to inline:

```c
int validate_mcb_chain(void) {
    union REGS r; struct SREGS s;
    r.h.ah = 0x52; intdosx(&r, &r, &s);  /* ES:BX -> List of Lists */
    unsigned seg = *(unsigned far *)MK_FP(s.es, r.x.bx - 2);
    for (;;) {
        unsigned char far *mcb = (unsigned char far *)MK_FP(seg, 0);
        if (mcb[0] != 'M' && mcb[0] != 'Z') return 1;  /* CORRUPT */
        if (mcb[0] == 'Z') return 0;                   /* end OK */
        seg += 1 + *(unsigned far *)(mcb + 3);
    }
}
```

Install `cleanup_audio()` via `atexit()` *and* via INT 23h (Ctrl-C) and INT 24h (critical error) handlers so it runs on abnormal termination as well. **Issue a full DSP reset on init too**, not only on exit — Quake's snd_dos.c sets the precedent and VOGONS confirms this fixes a class of "previous program left DMA dirty" issues, which is what would happen the second time HEARO ran after a previous crash.

## Conclusion: what changed and what to do next

The user's mental model was already largely right. The five facts that should reset priorities are: **(1)** PnP BIOS enumeration is small, well-spec'd, and the only path to YMF715 on the 320CDT — Linux's `opl3sa2.c` is your reference, and `YMH0021`'s 5th I/O port is the control base, encoded as `65 A8 00 21`; **(2)** the Watcom `__STK` issue is fixed by `-s -zu` plus `#pragma off (check_stack)` on a dedicated ISR translation unit, and Open Watcom's own `mouse.c` example documents the pattern; **(3)** the YMF715 is hostile in specific, enumerable ways (fake DSP 3.01, broken master-volume mixer, missing ADPCM, possibly unreliable 0xDA exit) — gate clone behavior on cmd 0xE3 copyright string, not version; **(4)** the OpenMPT wiki *is* the tracker bible, particularly the `Manual:_Compatible_Playback` page and the four `Development:_Test_Cases` index pages; aim for audibly identical, not bit-exact; **(5)** the "Cannot load COMMAND" wedge is almost certainly DMA-after-exit corrupting the MCB header of whatever paragraph DOS hands to COMMAND.COM — the fix is mask-DMA-controller before DSP-reset, and validate the chain before `AH=4Ch`.

The single highest-leverage next step is **implement the PnP BIOS enumerator and the OPL3-SA3 MISC probe together**, behind a unified API that returns `{ control_base, sb_base, wss_base, fm_base, mpu_base, irq, dma8, dma16 }`. That single piece of work unblocks SB Pro emulation initialization, lets HEARO log a complete hardware report on startup (which will help every subsequent debugging session), and brings HEARO to feature parity with what UNISOUND and SETYMF cannot do. Everything else — buffer allocation, exit cleanup, decoder correctness, defensive clone handling — has known recipes; this one was the actual gap.
